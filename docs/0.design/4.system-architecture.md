# 系统架构

## 进程架构

SkillForge 基于 Electron 的双进程架构，通过 electron-vite 统一构建三端代码。

```
┌──────────────────────────────────────────────────────────────┐
│                        Electron 应用                          │
│                                                              │
│  ┌────────────────────┐         ┌─────────────────────────┐  │
│  │   主进程 (Main)     │         │    渲染进程 (Renderer)   │  │
│  │   Node.js 环境      │         │    浏览器环境 (React)     │  │
│  │                    │         │                         │  │
│  │  - 应用生命周期      │ invoke  │  - 页面路由 (App.tsx)    │  │
│  │  - SQLite 数据库    │◄────────│  - Zustand 状态管理      │  │
│  │  - LLM API 客户端   │ IpcResult│  - 七巧板画布(FlowCanvas)│  │
│  │  - Agent 工具循环   │────────►│  - CanvasOp + undo/redo │  │
│  │  - 画布工具集       │         │                         │  │
│  │  - 文件系统操作      │ agent:  │  - 流式渲染/工具状态     │  │
│  │                    │ event   │  - 幽灵积木              │  │
│  │  AG-UI 事件发射 ────┼────────►│  - 事件订阅(sceneStore)  │  │
│  └─────────┬──────────┘  push   └────────────┬────────────┘  │
│            │                                 │               │
│            │       ┌──────────────────┐      │               │
│            └───────│  Preload 桥接层   │──────┘               │
│                    │  contextBridge   │                      │
│                    │  window.api      │                      │
│                    └──────────────────┘                      │
└──────────────────────────────────────────────────────────────┘
```

**双通道模式**：请求-响应走 `ipcMain.handle` + IpcResult 信封；智能体过程推送走 `agent:event`（webContents.send，AG-UI 事件），二者不混用。

## 主进程模块

### index.ts — 应用入口

- 创建 BrowserWindow（1400×900，minSize 1024×700）
- 启动顺序：initDatabase → registerIpcHandlers → createWindow
- 生命周期管理：window-all-closed / before-quit 时关闭数据库

### llm.ts — LLM API 客户端

- 支持 OpenAI / Azure / 自定义端点（OpenAI 兼容 chat/completions）
- `callLLMEx`：流式（SSE 逐 delta 回调）+ function calling + 退避重试（5xx/超时/网络错误 2s/5s/10s；**429 限流单独长退避 5s/21s/30s**）+ AbortSignal 中断
- `callLLM`：兼容版（仅返回文本，无重试无流式）
- 内置速率限制：MIN_INTERVAL = 1100ms；流式空闲超时按 chunk 重置
- 错误码：AUTH_ERROR / API_ERROR / EMPTY_RESPONSE / TIMEOUT / NETWORK_ERROR / ABORTED

### agentLoop.ts — Agent 多步工具循环（核心）

- `runAgentLoop`：循环「LLM 调用 → 执行工具 → 喂回结果」直到无工具调用或达到 maxSteps（默认 8）
- **原生模式**：function calling + 流式文本（TEXT_MESSAGE_* 事件）
- **降级模式**：模型不支持 tools 时自动切换 JSON 协议（提示词附加工具目录 + 严格 JSON 输出约定），按 `baseUrl|model` 缓存探测结果；降级模式非流式，完成后补发完整文本事件
- 工具可通过 `ctl.stop()` 提前结束本轮（如 ask_user 等待用户回答）
- 中断：`activeRuns` 注册表（runId → AbortController），`abortAgentRun(runId)`
- 全程发射 AG-UI 事件：RUN_STARTED / TEXT_MESSAGE_* / TOOL_CALL_* / RUN_FINISHED / RUN_ERROR

### canvasTools.ts — 画布工具集（智能体的"手"）

- `createCanvasTools(sceneId, runId, canvas, collector)` 工厂，绑定到一次 run
- 四个工具：`canvas_add`（用户确认内容直接上画布）/ `canvas_update` / `canvas_delete`（按 id 定位）/ `propose`（归纳推断 → 幽灵积木）
- 知识类型：工具枚举仅 **ACTIVE_TYPES**（flow/rule/insight，概念/关系暂缓）；`findEntry` 仍检索全部五类（兼容旧数据的修改/删除）
- 每次操作：改内存画布 → saveCanvas 落盘 → 推 CUSTOM canvas_update/proposal 事件 → 记入 collector（随 IPC 结果返回，渲染端幂等去重）

### agentEvents.ts — AG-UI 事件发射

- `emitAgentEvent(event)`：向所有窗口 `webContents.send('agent:event', event)`
- 事件类型定义共享于 `src/contracts/agent-events.ts`（基于 @ag-ui/core EventType）

### extraction.ts — 萃取智能体

- `runTurn`：构建系统提示（SKILL.md + 画布概要 + 覆盖度缺口摘要 + 参考文档）→ runAgentLoop（画布工具）→ 持久化对话 → 返回 reply + canvasUpdates + proposals
- **画布概要**（buildCanvasOutline）：紧凑视图（[id] 标题 | 证据等级 | 截断内容），代替整卡 JSON，供 agent 定位修改目标
- **缺口摘要**（buildGapSummary）：三类（flow/rule/insight）条目分布 + 缺失类别 + 探索性条目清单，驱动追问方向
- **长对话压缩**（buildConversationContext）：>24 条时较早历史摘要化（LLM 压缩，缓存于 preferences `convSummary:{sceneId}`，按消息数失效），保留最近 16 条原文
- `draftFromDocs`：从参考文档起草，agent 用 canvas_add 逐条建块（maxSteps 12）

### agents.ts — 引导和验证智能体

- `guideRunTurn`：runAgentLoop + 两个工具——`update_scene`（草稿持久化于 preferences `sceneDraft:{sceneId}`，实时推 CUSTOM scene_draft 事件）和 `ask_user`（设置 options + multiSelect 标记并 stop）；maxSteps 4；返回值含 optionsMultiSelect
- `getSceneDraft`：读取持久化场景草稿，供渲染端返回引导页时恢复（IPC `guide:getDraft`）
- `validationRun`：A/B 并行 callLLMEx，各带一路流式发射器（messageId 'bare'/'skill'）；差异分析解析为结构化 dimensions + overall（兼容旧 diffSummary 格式）；全程 RUN_STARTED/FINISHED/ERROR 事件
- 三个 agent 均通过 `resolveAgentLLMConfig(agentKey)` 取配置：agent_config 完整则用之，否则回落全局

### store.ts — 数据持久化

- SQLite（better-sqlite3，WAL，外键约束），路径 `{userData}/skillforge.db`
- 表：scenes / references / conversations / llm_config / llm_provider / agent_config / preferences
- preferences 通用读写：`getPreference` / `setPreference`（对话摘要缓存、场景草稿）
- Canvas JSON 文件：`{userData}/canvas/{sceneId}.json`（含 layout 布局）
- 参考文档文件：`{userData}/references/{sceneId}/`

### ipcHandlers.ts — IPC 中央调度

- 25 个 invoke 通道，命名 `{namespace}:{action}`，全部 wrapHandler 包装
- 命名空间：scenes / references / extraction / guide / validation / canvas / export / settings / **agent**（abort）
- 推送通道：`agent:event`（不走 invoke）

### errorHandler.ts / packager.ts / docParser.ts

- 同前：IpcResult 信封 / 健康检查 + ZIP 打包（experience-card.json 含 layout）/ 文档解析（20MB 上限）

## Preload 桥接层

通过 contextBridge.exposeInMainWorld 暴露 window.api，9 个命名空间：

| 命名空间 | 方法 | 职责 |
|----------|------|------|
| agent | onEvent（订阅，返回退订函数）/ abort | AG-UI 事件流 + 中断 |
| scenes | 5 | 场景 CRUD |
| references | 4 | 参考文档管理 |
| guide | 2 | 引导对话（runTurn）+ 草稿恢复（getDraft） |
| extraction | 2 | 萃取对话 + 文档起草 |
| validation | 1 | A/B 验证 |
| canvas | 1 | Canvas 全量更新 |
| export | 2 | 健康检查 + 打包 |
| settings | 8 | LLM/Agent 配置 |

## 渲染进程模块

### App.tsx — 页面路由

- 基于 Zustand currentPage 的 switch 路由（无路由库）
- 启动时 loadScenes() + **initAgentEvents()**（全局订阅 agent:event）
- 全局错误条（5 秒自动消失，用户中断不报错）

### sceneStore.ts — Zustand 全局状态

- 常规状态：scenes / currentScene / proposals / isLoading / error / llmConfig / llmProviders / currentPage / highlightedEntries / guideInput
- **事件流状态**：activeRunId / activeAgent / agentStatus / streamingText / liveSceneDraft
- **事件处理**（initAgentEvents）：文本增量累积、工具状态映射人话、CUSTOM canvas_update 实时应用（幂等）、proposal 去重入列、scene_draft 同步、RUN_FINISHED 清理；validate 双流绕行（由 Validate 页自行订阅）
- **CanvasOp 层**：`applyOpToCard` 纯函数 + `applyCanvasOp` / `undoCanvas` / `redoCanvas`（快照栈上限 50，切场景清栈），300ms 防抖持久化
- `applyProposal(id, position?, override?)`：提议实体化走 applyCanvasOp（支持位置与改写，可撤销）

### 页面与组件

| 组件 | 职责 |
|------|------|
| FlowCanvas | 七巧板画布：React Flow + KnowledgeNode（行内编辑）+ GhostNode（提议）+ 元件栏（三类）+ 连线 + undo/redo + 小地图；渲染兼容旧数据全部五类 |
| Conversation | 萃取对话：开场引导卡 + 流式气泡（Markdown）+ 工具状态 + 中断按钮 + 提议数提示 |
| Markdown | 轻量 Markdown 渲染（零依赖）：加粗/行内代码/分隔线/标题/列表/表格，未识别语法纯文本兜底 |
| Guide | 引导页：流式（Markdown）+ 选项芯片（单选/多选/其他行内输入）+ 属性面板实时同步（liveSceneDraft）+ 草稿恢复（guide:getDraft）+ 友好错误气泡 |
| Validate | A/B 双流渲染（订阅 agent:event 按 messageId 分流）+ 维度卡片 |
| ReferencePanel | 参考文档：上传/勾选/删除 + 从文档起草 + 收起回调（onCollapse） |
| ConceptTree | 概念树，随概念/关系类型暂缓而下线，保留备查 |
| Canvas（旧） | 分区卡片列表，已被 FlowCanvas 替换，保留备查 |

## 数据流

### 萃取对话数据流（双通道）

```
用户输入 → sceneStore.runTurn()
  → window.api.extraction.runTurn(sceneId, message)   [invoke 通道]
    → extraction.runTurn()
      → 构建系统提示（SKILL.md + 画布概要 + 缺口摘要 + 文档）
      → runAgentLoop(画布工具)
          每步: callLLMEx(流式) ──┐
            ├─ TEXT_MESSAGE_CONTENT ─────► [agent:event 推送通道]
            ├─ TOOL_CALL_START/RESULT ───►       │
            └─ 工具执行: 改画布+落盘            │
                 └─ CUSTOM canvas_update ──►     │
                                                 ▼
                                    sceneStore 事件处理:
                                      流式气泡 / 工具状态 / 画布实时上块+高亮 / 幽灵积木
      → 持久化对话消息
    ← IpcResult<RunTurnResult>（reply + canvasUpdates + proposals）
  ← handleIpc() 解包 → 最终消息入列，画布幂等合并，提议去重
```

### 用户画布操作数据流

```
拖拽/编辑/连线/采纳提议 → applyCanvasOp(sceneId, op)
  → 快照入 undo 栈 → applyOpToCard 纯函数 → set(currentScene.canvas)
  → schedulePersist（300ms 防抖）→ window.api.canvas.update → saveCanvas JSON 落盘
```

### IPC 信封模式

```
渲染进程                          主进程
  │  window.api.xxx(args)          │
  │  ──────────────────────────►   │  wrapHandler 自动 try/catch
  │  ◄──────────────────────────   │
  │  IpcResult<T>                  │  { success: true, data } | { success: false, error }
  │  handleIpc() 解包：成功返回 data，失败抛 Error（ABORTED 静默）
```

## 存储架构

### SQLite 数据库（skillforge.db）

| 表 | 主键 | 核心字段 | 说明 |
|----|------|----------|------|
| scenes | id (TEXT) | name, status, created_at, updated_at | 场景元数据 |
| references | id (TEXT) | scene_id (FK), filename, stored_path, extracted_text, include_in_package | 参考文档 |
| conversations | id (TEXT) | scene_id (FK), role, content, created_at | 对话消息 |
| llm_config | id (=1) | provider, api_key, model, base_url | 全局 LLM 配置（单行） |
| llm_provider | id (TEXT) | name, base_url, api_key, models (JSON), sort_order | Provider 列表 |
| agent_config | agent_key | provider, api_key, model, base_url | 各 Agent 独立配置 |
| preferences | key (TEXT) | value | 通用 KV：`convSummary:{sceneId}`、`sceneDraft:{sceneId}` 等 |

索引：idx_references_scene / idx_conversations_scene

### 文件系统

| 路径 | 内容 |
|------|------|
| {userData}/canvas/{sceneId}.json | ExperienceCard JSON（五类知识 + layout 布局） |
| {userData}/references/{sceneId}/{filename} | 参考文档文件副本 |

### 设计决策

- Canvas 数据不存 SQLite：ExperienceCard 结构灵活、大小不确定，JSON 文件更适合
- layout 为可选字段：旧数据自动排版，向后兼容
- 智能体不回吐整卡 JSON：增量工具调用（省 token、防丢条目），画布概要 + id 定位
- 提议（proposals）不持久化：会话内待确认状态，采纳后才成为正式条目
- 幽灵积木位置不入经验卡：本地暂存，采纳时随实体化带入 layout
- 概念/关系暂缓采用"入口收窄、数据保留"策略：工具枚举/元件栏/缺口摘要只放开三类，类型定义、存储结构、旧数据渲染全部保留——将来启用零迁移
