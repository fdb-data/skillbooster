## 项目结构

```
SkillForge/
├── electron/main/          # 主进程（Node.js 环境）
│   ├── index.ts            # 应用入口
│   ├── llm.ts              # LLM 客户端（流式/tools/重试）
│   ├── agentLoop.ts        # Agent 多步工具循环
│   ├── canvasTools.ts      # 画布工具集（agent 的"手"）
│   ├── agentEvents.ts      # AG-UI 事件发射
│   ├── extraction.ts       # 萃取智能体
│   ├── agents.ts           # 引导/验证智能体
│   ├── store.ts            # SQLite + 文件持久化
│   ├── ipcHandlers.ts      # IPC 中央调度
│   ├── errorHandler.ts     # IpcResult 信封

│   └── docParser.ts        # 文档解析
├── electron/preload/       # 预加载桥（contextBridge）
├── src/                    # 渲染进程（React + TypeScript）

│   ├── components/         # FlowCanvas / Conversation / Markdown / ReferencePanel …
│   ├── utils/              # 工具函数
│   └── styles/             # tokens.css + global.css
├── resources/              # Agent 提示词（打包进应用）
│   ├── experience-extraction/SKILL.md
│   └── agents/guide.md, validate.md
├── tests/                  # vitest：tests/main/ + tests/renderer/ + setup.ts
└── build/                  # electron-builder 资源
```

## 编码规范

### TypeScript

- strict 模式全局启用，零 any 容忍
- 项目引用分离：tsconfig.node.json（主进程）/ tsconfig.web.json（渲染进程）
- 路径别名：`@/` → `src/`
- 索引 ExperienceCard 的五类数组必须用 `KnowledgeKey`（`'flows' | 'rules' | ...`），**禁止** `keyof ExperienceCard`（会撞上 layout 字段）

### 知识类型启用范围

- 当前版本只启用 **flow / rule / insight** 三类（concept/relation 为企业级特性暂缓）
- 新增/提议入口（工具枚举、元件栏、缺口摘要）只放开三类；类型定义、存储结构、旧数据的渲染与修改删除保持全部五类兼容
- 不要删除 concepts/relations 字段或其类型定义——将来启用要零迁移

### 进程隔离

- 渲染进程**禁止**直接使用 Node.js API，必须通过 preload contextBridge
- 主进程**禁止**直接操作 DOM
- 所有跨进程通信走 IPC，不使用 remote 模块

### IPC 规范

**请求-响应**（invoke）新增三步同步完成：

1. **ipcHandlers.ts**：注册 handler，使用 wrapHandler 包装
2. **preload/index.ts**：通过 contextBridge 暴露方法
3. **global.d.ts**：添加 Window.api 类型声明

所有 invoke 返回值必须为 `IpcResult<T>` 信封；渲染进程通过 `handleIpc()` 解包，失败抛 Error。不允许在 handler 中直接 throw 出信封之外。

**事件推送**（agent:event）：

- 主进程只通过 `emitAgentEvent()` 发事件，事件类型必须在 `src/contracts/agent-events.ts` 的 `AgentEvent` 联合类型中
- 事件命名遵循 AG-UI 标准（@ag-ui/core EventType）；SkillForge 扩展一律走 CUSTOM 事件（name + value）
- 渲染进程统一在 sceneStore.initAgentEvents 订阅；页面级特殊需求（如 Validate 双流）自行订阅并按 runId/messageId 过滤
- 过程推送走事件，最终结果走 invoke 返回——двa通道职责不混

### Agent 开发规范

- 新增 agent 能力一律通过 **runAgentLoop + 工具**实现，不写裸 LLM 调用 + JSON 解析
- 工具定义（LLMToolDef）的 description 和参数 JSON Schema 用中文写清语义和边界——它们就是提示词的一部分
- 工具执行器必须返回结果字符串（成功描述或 `错误：...`），不抛异常给循环
- 画布类工具走 canvasTools 工厂，保证「落盘 + 事件 + collector」三件套一致
- 需要等待用户输入的工具调用 `ctl.stop(finalReply)`
- agent 配置用 `resolveAgentLLMConfig(agentKey)`，不直接 getLLMConfig

### LLM 调用规范

- 一律用 `callLLMEx`（流式/tools/重试）；`callLLM` 仅为兼容保留
- LLM 返回的 JSON 必须容错解析（兼容 markdown code fence 包裹）
- 内置速率限制：最小间隔 1.1s，不可绕过
- 遇到超时或 429 依赖内置退避重试，不可放弃或换方案；429 走专用长退避（5s/21s/30s，适配按分钟限速的供应商），其余可重试错误走 2s/5s/10s
- 默认参数：temperature 0.7, max_tokens 4096, timeout 120s（验证 60s）

### 画布操作规范

- 渲染进程改画布数据**只走 `applyCanvasOp`**（add/update/delete/move/connect/disconnect），不直接拼 canvas 对象——否则丢 undo、丢持久化、丢布局清理
- `applyOpToCard` 保持纯函数，可独立测试
- 智能体改画布走主进程 canvasTools；渲染端事件应用必须幂等（add 重复 id 视为替换）
- 删除条目必须级联清理 layout.positions 和 layout.edges

### 数据持久化规范

- Canvas 数据（ExperienceCard 含 layout）→ JSON 文件：`{userData}/canvas/{sceneId}.json`
- 结构化元数据 → SQLite：scenes / references / conversations / config 表
- 会话级缓存（对话摘要、场景草稿）→ preferences 表，key 格式 `{用途}:{sceneId}`
- 参考文档 → 文件副本：`{userData}/references/{sceneId}/{filename}`
- SQLite 使用 better-sqlite3，WAL 模式，外键约束启用

### 错误处理规范

- 主进程 handler 使用 `wrapHandler` 包装，自动 try/catch 返回 IpcResult
- 渲染进程通过 `handleIpc()` 解包，失败设置 store.error
- 全局错误条显示 store.error，5 秒自动消失
- 引导对话失败不允许静默：对话区内显示友好错误气泡（friendlyError，429 限流给出等待指引）
- 用户主动中断（LLMError code=ABORTED，message='已中断'）静默处理，不显示错误
- agent run 出错必须发 RUN_ERROR 事件再 rethrow（保证渲染端状态清理）

## 命名规范

### IPC 通道

- 格式：`{namespace}:{action}`
- invoke 命名空间：scenes / references / extraction / guide / validation / canvas / export / settings / agent
- 推送通道：`agent:event`（常量 AGENT_EVENT_CHANNEL）

### 文件命名

- 主进程：camelCase（agentLoop.ts / canvasTools.ts / agentEvents.ts …）
- 渲染进程：PascalCase 组件（FlowCanvas.tsx / Conversation.tsx …）
- 类型文件：kebab-case（ipc-types.ts / agent-events.ts / card.schema.json / global.d.ts）
- 样式：kebab-case（tokens.css / global.css）

### 数据库

- 表名/列名：snake_case；TypeScript 接口：PascalCase，字段 camelCase

## 测试规范

### 测试框架

- Vitest + @testing-library/react + jsdom/happy-dom
- 测试目录：`tests/main/` + `tests/renderer/`
- setup 文件：`tests/setup.ts`（mock electron（含 BrowserWindow.getAllWindows）/ better-sqlite3 / fs / window.api（含 agent.onEvent）/ ResizeObserver）

### Mock 约定

- 主进程测试 mock `callLLMEx` 返回 `{ content, toolCalls }`；模拟工具流程用 mockResolvedValueOnce 链（先 toolCalls 后纯文本）
- store mock 必须含 `resolveAgentLLMConfig` / `getPreference` / `setPreference`
- 渲染组件测试 mock useSceneStore selector 模式，mockStore 需包含事件流字段（streamingText / agentStatus / activeRunId / proposals …）
- `tests/setup.ts` 的 window.api mock 需与 preload 同步维护（如 guide.runTurn/getDraft）——新增 IPC 后记得补 mock

## 安全规范

### Electron 安全

- contextIsolation: true / nodeIntegration: false / sandbox: true / webSecurity: true

### CSP 策略

```
default-src 'self'
script-src 'self' 'unsafe-eval'
style-src 'self' 'unsafe-inline'
connect-src 'self' https://*.openai.com https://*.anthropic.com https://*.azure.com http://localhost:*
img-src 'self' data:
```

注：LLM 调用全部在主进程发起，渲染进程不直连任何 LLM 端点。

- API Key 存储在 SQLite，不写入日志；UI 中掩码显示
- 参考文档复制到本地后独立存储，不依赖外部路径

## Agent 提示词管理

### 文件位置

- 萃取：`resources/experience-extraction/SKILL.md`（v2.0，工具协议）
- 引导：`resources/agents/guide.md`（意图分析 + update_scene/ask_user）
- 验证：`resources/agents/validate.md`（五维度结构化输出）

### 加载机制

- 生产模式：process.resourcesPath + '/resources/'；开发模式：项目根目录 resources/
- Settings 页可打开提示词文件查看/编辑

### 输出协议

- 萃取/引导 agent：**工具调用**（原生 function calling 或降级 JSON 协议，由 agentLoop 统一处理），提示词只写策略不写工具 schema
- 验证 agent：JSON `{ "dimensions": [{name, analysis}...], "overall": "..." }`

## 技术栈锁定

以下技术选型已锁定，不允许更换：

| 技术 | 锁定原因 |
|------|----------|
| better-sqlite3 | 同步 API 性能优，WAL 可靠，原生模块已处理 asarUnpack |
| Electron / electron-vite | 桌面框架与构建唯一选型 |
| React + TypeScript / Zustand | 渲染进程唯一选型 |
| @xyflow/react | 画布唯一选型 |
| @ag-ui/core | 事件协议唯一选型（仅事件类型层，不引入 CopilotKit） |

遇到编译/运行问题，必须排查根因解决，不允许绕道换方案。
